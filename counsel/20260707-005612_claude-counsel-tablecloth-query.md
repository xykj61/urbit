## The Name Carries Two Meanings, and They Have Never Met

Before this sitting reaches for a shape, it is worth saying plainly what a search through the tree's own record turned up: **Tablecloth already means two different things, seated in two different places, and nothing has yet reconciled them.**

`context/LEXICON.md` holds both entries side by side — one line calling it *"application data store (horizon — role still shaping),"* a later line calling it *"layout or structured-view layer; horizon."* The older thread, from `20260620.043812`'s split brief, is precise and different again: *"Tablecloth is the general-purpose content-addressed store: an accreting collection of immutable blobs, each named by the hash of its content... Brix is the composing language."* In that telling, Tablecloth holds build artifacts for Brix to compose, and "query" would mean looking up a blob by its hash or its metadata — nothing to do with the referential namespace this whole arc has grown, which did not yet exist when that brief was written.

The newer thread lives in the namespace brief's own foundational paragraph, seated the same day the namespace vocabulary was: *"query, when its season finally comes, is Tablecloth — held back deliberately, because the namespace postpones being a database until it must."* Read in place, that sentence is unambiguous about what it means by the word — it is talking about `recall`, `bolt`, and everything this arc just built, not about Brix's artifact store.

This counsel takes the namespace brief's own words as the operative meaning for the question actually in front of it, because that is the document governing the Q3 list this ruling itself ordered, and because every item on that list through this one has been a namespace capability. The older, Brix-facing thread is not wrong, and it is not this counsel's to resolve — it is simply a different thing wearing the same name, worth a line in whichever revitalization pass eventually touches the oldest active-designing briefs, rather than a question this sitting needs to settle to answer what was actually asked.

## The Smallest Honest Ring, Given That Reading

`recall_by_mark.rye` already reads by Tilak, and it already refuses cleanly rather than guessing when a mark is shared. Its query today requires peer, bolt, and revision all at once, narrowing to a single leaf or refusing as ambiguous. The whole of Tablecloth query's first ring is loosening exactly those three fields from required to optional, and changing what happens when more than one leaf matches from *refuse* to *return every one, in the order held* — nothing else. `BoltCatalog`'s own leaves are a flat array bounded at sixteen total, so a query that omits revision walks every revision of a bolt, one that omits bolt walks every bolt a peer holds, and one that omits all three walks the whole small catalog — each of these is the same loop over the same array recall_by_mark already runs, filtering on fewer fields, never more than sixteen results, never a new index, never a store. This is the namespace staying exactly as far from being a database as the foundation brief already promised it would.

**Witness shape, sketched:** a small bolt with a few leaves spanning two revisions and two marks is enough — the same shape recall_by_mark's own witness already uses, extended by one more leaf at a second revision. A query naming peer and bolt with revision left open should return leaves from both revisions; a query naming only a Tilak should return every leaf wearing it regardless of which bolt or revision holds it; a query matching nothing should return an empty result rather than any refusal, the same honesty about absence `resolveBinding`'s `never` outcome already carries. Recall by exact name, recall by mark within one revision, and this looser query should all agree completely on any leaf all three could reach — the same convergence proof this arc keeps returning to, asked of one more surface.

## What This Does Not Yet Claim

Sixteen leaves is the whole catalog today, so nothing about this ring says anything about performance at a scale the namespace has never been asked to hold, and nothing about it invents a query language, a plan, or an index — it is a filter with fewer required fields, walked once, bounded by a constant that already exists. Whether the namespace ever needs more than that — pagination, a real index, a store shaped like the older Brix-facing Tablecloth — stays a question for whenever `max_bindings` itself is asked to grow, which is not a question this ring needs to answer to be honest about what it does today.
