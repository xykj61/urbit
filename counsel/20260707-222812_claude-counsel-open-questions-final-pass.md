## One Question Resolved Outright — Kumara Has Nothing to Graduate Yet

TASKS names Kumara's module home as open, framed the same way Weave's and Dexter's graduation questions were framed — as if the question were *when* a threshold is crossed. It checked differently. A tree-wide search for the name in any authored `.rye` file returns nothing at all: no `kumara.rye`, no file carrying the name anywhere. Every place this tree currently does identity work — signing a fact, generating a deterministic keypair, verifying a signature — reaches directly for `std.crypto.sign.Ed25519`, scattered across as many files as need it, with "Kumara" living only in prose, as the name for the *pattern* those scattered calls all quietly share. The graduation rule asks whether a second consumer depends on an aspect's surface. Kumara does not yet have a first consumer in that sense, because it does not yet have a surface — there is no file to consume. This is not a graduation question at all; it is an extraction question, one stage earlier, and it has an honest answer available without anyone's word: **Kumara's home stays open because Kumara has not yet been built**, and the real open question, if this bench's finding is right, is whether the scattered `Ed25519` calls are worth centralizing into one small file the day a second one needs the exact same helper — which is Gall's Law's own test, not a naming ceremony.

## The Two-Backend Seam, Leaned On Rather Than Decided

The hosted Wayland path already runs — this bench has built and run `wayland_seed.rye` directly, more than once, across this whole engagement. The open question is whether a second, sovereign backend — Brushstroke drawing straight to a framebuffer under QEMU, with no host compositor in the loop at all — earns a build now or waits. This bench's honest lean: wait, on the same grounds that have governed everything else this arc has touched. The vision itself names x86_64 hosted as today's ground and RISC-V bare metal as the horizon, not the present target; Aurora's own bare-metal work is real, still a boot relay rather than yet a display consumer; and a second backend built before anything actually needs one is exactly the shape of frontier-funded-ahead-of-proven-ground this tree has declined every other time the question came up — the compiler fork, PRNG and `parse_int` before a real call site, `no_padding` before a real `extern struct`. Build the sovereign sibling the day Aurora's own path genuinely wants a picture on a screen with no compositor to ask; not before. This is a lean, not a ruling — the taste question of *how soon is too soon* is fairly Kaeden's, even where the reasoning is this bench's own.
