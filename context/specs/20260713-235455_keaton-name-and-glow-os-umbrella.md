## Two Decisions, Sealed Here

### 1. The name is Keaton

**Keaton is the real name. "Kaeden" was a personal pseudonym.** From this point forward this is a professional work project — one that Keaton Dunsford (`xykj61`) intends, in time, to propose to Tlon Corporation as a paid contractor under legally signed documents. Every reference to "Kaeden" anywhere in this tree — in dated session logs, in living rules, in commit-message bodies — denotes Keaton Dunsford. This document is the one place that says so plainly, so no future reader mistakes the pseudonym for a second person.

**How the transition is applied, per this fork's own law:**

- **Living files migrate on touch.** The collaboration rule already governs renames: "migrate names opportunistically when you touch a file — rather than renaming silently across the tree in one sweep." Living rules, READMEs, and specs move to "Keaton" as they are next edited. `PUBKEYS.md` already reads Keaton Dunsford; the session-logs and acme-employee-voice rules are updated alongside this marker.
- **Dated artifacts stay as written.** Session logs and dated specs stamped before this marker keep "Kaeden" exactly as recorded, per the one-clock naming law and `dated_guard`. They are the honest record of the work as it happened under the pseudonym. This marker tells the reader how to read them.
- **Commit history is not rewritten.** The git *author* of every commit in this fork is already `Keaton Dunsford <keatondun@gmail.com>` — verifiable with `git log --format='%an'`. Only some commit-message *bodies* say "Kaeden," and those are dated artifacts sealed by this marker. Rewriting them would require a force-push of a 35,000-commit history to two forges and would contradict the fork's own "accrete, never break" discipline for no benefit the author field does not already provide. **We do not rewrite the pushed history.** If Keaton ever wants it anyway, it is a deliberate, separate, explicitly-approved force-push — not a default.

### 2. The umbrella is Glow OS

**"Glow OS" is the new name for what was "Rye OS"** — the whole system, by Keaton Dunsford / `xykj61`, carrying the same CONTRIBUTING discipline adapted from Urbit / Tlon. The naming hierarchy, stated once, clearly:

| Layer | Name | Note |
|---|---|---|
| The language | **Glow** | the fused Rye-Hoon language (accepted `20260713.225658`) |
| The umbrella OS | **Glow OS** | was "Rye OS"; the whole project |
| The four OS variants | **Reya · Riyo · Trey · Triz** | all written in Glow; all part of Glow OS; **all four confirmed** `20260714.035600` as two diverse-redundant pairs — Riyo/Reya and Trey/Triz |
| The shell | **Rishi** (proposed: **Dojo**, or a new Glow-OS name) | still to confirm — see the names spec |
| The style/law | **TAME Guidance** | one document, governs all of Glow OS |

**The mass rename Rye OS → Glow OS is deferred, not done here.** "Rye" carries 1,474 inbound references (per the naming-mapping proposal); a sweep of that size waits for Keaton's explicit word and goes cheapest-first, exactly as every other proposed rename in this fork. This marker records the *decision*; it does not execute the *migration*. Until the migration runs, "Rye OS" in existing files reads as "Glow OS" the same way "Kaeden" reads as "Keaton."

## Why It Is Shaped This Way

- A professional project that will be proposed for payment wants its real name on it — so the name is recorded plainly, once, authoritatively.
- The fork already has a working, honest discipline for exactly this kind of transition (the veganreyklah2 → fork new-journey marker at `20260713-222416`). This marker is its sibling: seal the past as the record of the past, carry the new name forward, never pretend the past was other than it was.
- Rewriting history to erase a pseudonym would be the one move that contradicts every immutability rule this fork has just spent a day establishing. The marker is the mature alternative.
