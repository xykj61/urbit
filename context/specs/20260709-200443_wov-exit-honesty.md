## 1. What This Spec Is, and What It Is Not

**Exit honesty** is WOV's safety promise, written before any overlay code:

> With each root fact, the instance publishes enough for any holder to **prove their balance at that root** and **exit to MALA without the monarch's cooperation going forward**.

A throughput lane whose users cannot leave is a cage. A lane whose exits are facts is a gift.

**This document** names the facts, the proofs, the exit path, and the first-lap bounds. Lap 1 implements an **in-memory** book (no live TigerBeetle client yet) with golden bundle, root fact, prove, exit, and double-exit refuse.

**WOV itself** (reminder): a Rye overlay on **pinned** TigerBeetle in `gratitude/tigerbeetle`; each instance an L2 book; conservation by the ledger's invariants; anchored to MALA by monarch-signed root facts. One or two monarchs; never three (three is Mycelium at M4).

## 3. Root Fact — What Must Be Published

Each root is a **MALA-shaped signed fact** appended to the local MALA log (same `receipt_core` field grammar: `from` / `to` / `amount` / `memo` / `stamp`), with memo prefix `wov:root`.

### 3.1 Required memo payload (after the prefix)

Canonical, space-separated tokens inside `memo` (stable order):

```text
wov:root <instance-id-hex32> <seq-u64> <state-root-hex64> <accounts-digest-hex64> <prev-root-hex64>
```

| Token | Meaning |
|-------|---------|
| `instance-id-hex32` | 16-byte instance id, hex (stable for the life of the book) |
| `seq-u64` | Monotonic root sequence; starts at `1` |
| `state-root-hex64` | SHA3-256 of the **exit bundle** bytes for this root (see §4) |
| `accounts-digest-hex64` | SHA3-256 over the sorted account table in the exit bundle |
| `prev-root-hex64` | Prior root's `state-root-hex64`, or 64 zero hex at genesis |

### 3.2 Fact field conventions (lap 1)

| Field | Value |
|-------|-------|
| `from` | Monarch public key hex (64) |
| `to` | Same as `from` for a solo monarch; second monarch's key when dual-control is live |
| `amount` | `0` (root facts move no MALA units by themselves) |
| `memo` | Payload above |
| `stamp` | One-clock `YYYYMMDD.HHMMSS` |

Dual control (two monarchs): both signatures appear as two consecutive log lines over the **same** canonical fact bytes, or a later lap names a two-sig envelope. Lap 1 may stay solo-monarch.

### 3.3 Cadence

First lap: **at least one root per witness run** (not wall-clock hourly yet). Horizon: hourly roots as the July 2 gate sketched. Cadence is operational; honesty is structural.

## 4. Exit Bundle — Enough to Leave

Beside each root fact, the monarch publishes an **exit bundle**: a content-addressed resin (Amber/Tilak grammar when sealed; plain `.bron` fixture for the first witness) whose SHA3-256 equals `state-root-hex64`.

### 4.1 Bundle contents (minimum)

1. **Header** — `wov-exit-bundle/v1`, `instance-id`, `seq`, `stamp`
2. **Account table** — every account with non-zero balance at this root, sorted by account id ascending:

   ```text
   <account-id-hex64> <balance-i64-decimal> <nonce-u64>
   ```

   `account-id` is the holder's Kumara public key hex (same width as MALA `from`/`to`).
3. **Conservation line** — sum of balances equals issued-into-WOV minus redeemed-from-WOV at this root (both totals listed).
4. **Monarch attestation** — Ed25519 signature over the bundle body (may match the root fact signer).

### 4.2 What a holder proves

Given the exit bundle and the signed `wov:root` fact on the MALA log:

1. Verify the root fact signature(s) against the known monarch pubkey(s).
2. Recompute SHA3-256 of the bundle; match `state-root-hex64`.
3. Find their `account-id` row; read `balance` and `nonce`.
4. Check conservation line arithmetic.

That is the **balance proof at the last root**. No live query to the monarch is required after the bundle and root fact are in hand.

## 5. Exit to MALA — Without Further Monarch Help

**Exit** means: the holder's WOV balance at root `N` becomes a MALA credit on the issuer log, and that WOV balance must not be spendable again on the L2 book.

### 5.1 Exit fact (holder-initiated)

A holder who already holds the exit bundle for root `N` posts a MALA fact:

```text
memo: wov:exit <instance-id-hex32> <seq-u64> <balance-i64> <nonce-u64> <state-root-hex64>
```

| Field | Value |
|-------|-------|
| `from` | Holder pubkey hex |
| `to` | MALA issuer pubkey hex |
| `amount` | The proven balance (positive decimal) |
| `memo` | Payload above |
| `stamp` | Holder's stamp |

### 5.2 Issuer acceptance rules (deterministic)

The MALA issuer (or a verifier folding the log) accepts `wov:exit` when **all** hold:

1. A `wov:root` for that `instance-id` and `seq` already sits on the MALA log, signatures valid.
2. `state-root-hex64` matches that root fact.
3. The cited exit bundle (retrieved from local store or Amber) hashes to that root and lists this holder with that `balance` and `nonce`.
4. No earlier accepted `wov:exit` for the same `(instance-id, account-id, seq)` exists (one exit per account per root).
5. Optional lap-1 pin: only the **latest** published root may be exited against (simpler; later laps may allow exit against any published root with freeze rules).

On accept, the issuer appends the exit fact (and may mint/send MALA units per existing MALA fold rules so the holder's MALA balance increases by `amount`). The WOV book, when it next sees the MALA log, treats that account's L2 balance as **redeemed** at that root and refuses further L2 spends that would double-exit.

### 5.3 Monarch unavailability

If the monarch never answers again, holders who already fetched root `N`'s exit bundle still complete steps §5.1–§5.2 against the MALA issuer alone. Holders who never fetched a bundle cannot prove a later balance — that is why **publishing** the bundle with every root is the honesty duty, not an optional export.

## 6. Threats This Spec Answers

| Threat | Answer |
|--------|--------|
| Monarch goes silent | Prior published bundles + root facts suffice for exit |
| Monarch withholds one holder's row | Conservation + full account table make omission detectable at verify time |
| Double exit | `(instance-id, account-id, seq)` uniqueness on the MALA log |
| Forged root | Kumara signatures on root facts; pinned monarch pubkeys |
| Cage (fast lane, no leave) | Exit path is a first-class fact shape, not a support ticket |

### What Stays Outside This Spec

- Byzantine agreement among three+ monarchs (Mycelium / M4)
- Hiding balances from the exit bundle (privacy is SLC-L4's room; exit honesty prefers auditable tables in lap 1)
- Live TigerBeetle client protocol details (first code lap)
- Cross-instance atomic swaps
