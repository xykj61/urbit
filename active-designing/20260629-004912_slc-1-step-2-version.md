## The Problem, Plainly

Step 1 closed its circle: `rishi repl` reads a line, evaluates it in process, carries the bindings forward, and ends cleanly on `:quit`. A person can sit at the prompt and work. Yet the moment that session ends, the work it held dissolves. Nothing of the session reaches Mantra's content-addressed store, so a day's exploration leaves no fact behind that another tool — or tomorrow's self — can read back and trust.

Step 2 closes that second gap with one verb. The session gains a durable home in the store it already lives beside: the lines a person typed, woven into Mantra's history as an honest, content-named fact, advancing HEAD the way any other `mantra add` does. History comes to rest where the version layer can already see it.

## What Version Is

**Version** is the third of SLC-1's four verbs — type, run, **version**, recall — and it does exactly one thing: it persists the session into Mantra's store on request.

| Word | Meaning for this verb |
|------|------------------------|
| **Simple** | One meta-command, one session file, one `mantra add` — the store layer Mantra already carries does the weaving |
| **Lovable** | Your session becomes a fact you can keep, on your own metal, named by its own content |
| **Complete** | `:version` writes the transcript and advances HEAD; the loop can be saved, every time you ask |

Version grows from the proven Step 1 loop and the Mantra seed already green. It adds memory to a session that until now lived only in the moment.

## The Two Structures

Step 2 introduces two small structures, each bounded on purpose, each doing one thing. Keeping them apart is what keeps the verb simple: one structure serves recall, the other serves the weave, and neither reaches into the other's work.

### The session buffer — the cache for recall

A bounded, in-memory record of the input lines a person has typed, most recent kept, oldest released. It exists to serve `:history` now and `:recall` in Step 3. It mirrors the bounded shape `Env` already keeps — a fixed array, a `u32` count, an assert that the count never passes capacity, an edge met loudly rather than by silent overwrite.

```zig
const max_history: u32 = 50; // input lines the session buffer holds; the metal may teach a truer number
```

The lap is a cache. It holds the recent past for the hand to reach back into, and it forgets gently past its bound. The durable record lives elsewhere.

### The transcript — the durable record

A bounded, in-memory text record of the session as it unfolds: the prompt, the input line, and any error the evaluator named, gathered in order. On `:version` it is written whole to `.mantra/session.log` and handed to Mantra. The transcript is richer than the lap — it carries prompts and errors, not lines alone — and it is the thing the weave remembers.

```zig
const max_transcript_bytes: u32 = 65536; // the session's durable text, bounded; a first estimate the metal can revise
```

Holding the transcript in memory until `:version` keeps the discipline clean in two ways at once. Memory stays bounded by a named ceiling, the way TAME asks of every growing thing. And the store stays unsurprising: `.mantra/` comes into being only when a person asks to keep the session, never unbidden from a single typed line. When the transcript nears its bound, the loop says so plainly and invites a `:version` to checkpoint, rather than wrapping in silence.

A growing append-on-disk transcript, for sessions that would outrun the in-memory bound, rests as a clean horizon. The bounded in-memory record is the simple, complete choice for this lap.

## The Composition — Rishi Drives Mantra

History lives in Mantra's store, and Rishi reaches that store the honest way: by **driving the Mantra seed as a separate program**, never by folding Mantra's internals into the shell. This is composition over absorption — the same discipline that lets each module teach its one function — and it costs nothing extra, because Rishi already runs external programs through `doRun`'s `std.process.run` path and reads back a four-field result of `out`, `err`, `code`, and `ok`.

On `:version`, the loop moves in three plain steps:

1. **Flush.** Write the in-memory transcript whole to `.mantra/session.log`, so Mantra reads a complete file.
2. **Ensure the store.** On the first `:version` of a session, check for `.mantra/`; when it is absent, run `mantra init`. A stat first keeps repeated versions quiet.
3. **Weave.** Run `mantra add .mantra/session.log`. Mantra computes the diff against its current weave, writes the new blob by its SHA3-256 name, writes the commit, and advances HEAD. The loop reports the new HEAD back to the prompt.

**Resolving the Mantra binary** follows the settled ruling: the loop reads an optional `RISHI_MANTRA` environment variable for the binary's path, and when it is unset, it calls `"mantra"` resolved on `PATH`. The witness controls the path it needs; the loop stays portable to any machine that carries Mantra.

The only Mantra commands this verb requires are `init` and `add`. The fuller Mantra surface — `log`, `status`, repository-wide history — stays out of scope; that is Mantra's own module, not this lap.

## The Meta-Commands Version Adds

Two meta-commands arrive with this step. Both begin with `:`, keeping session control distinct from Rishi surface syntax, so the language parser stays exactly as small as Step 1 left it.

- **`:history`** — print the numbered entries of the session buffer, most recent context near at hand.
- **`:version`** — ensure the store, write the transcript, weave it through `mantra add`, and report the advanced HEAD.

`:recall` arrives in Step 3 and reads from the same session buffer this step fills. `:quit` and `:q` carry forward unchanged. A `:` line that matches no known command reports an unknown meta-command and the loop continues, exactly as Step 1 already handles it.

## What the Transcript Holds

The transcript records what the loop already holds centrally, and claims no more. Per the settled ruling, a v1 record carries the **prompt**, the **input line**, and any **error the evaluator named** — all three available in one place in the loop today, with no new plumbing.

Capturing the **stdout** of each builtin into the transcript would mean threading a writer through every `do*` helper, and that re-plumbing rests as a clean horizon rather than a cost this lap pays. The transcript of inputs and errors is a coherent, complete record of what a person did and where the loop spoke back about trouble — honest at its scope, and woven faithfully into Mantra's history.

## Where History Lives

```
<cwd>/
  .mantra/
    HEAD              # 64-char hex name of the current commit blob, + newline
    blobs/            # content-addressed store, SHA3-256 names
    session.log       # the transcript Rishi writes; mantra add weaves it
```

The store appears on the first `:version`, created lazily through `mantra init`. The session buffer and the transcript live in memory until then. The weave in `blobs/` is the durable record; the in-memory lap is the cache for the hand.
