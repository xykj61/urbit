## The Blocker, Named at Its Door

The handoff reported a symptom that pointed one way while the fault lived another way. The console showed a TX guest that reached `driverOk`, wrote `QUEUE_NOTIFY`, and then spun forever with `used.idx` stuck at zero. That reading invited a queue-programming hunt, and the six questions followed it faithfully. The close reading on metal tells a cleaner story: the queue programming carried two real errors worth fixing, and the deeper reason the guest hung sat entirely outside virtio — in the linker script that lays a freestanding stage into memory. This ruling answers all six questions with the queue path settled, then names the true blocker so the next stage inherits a boot that holds.

## Question One — The Canonical Queue Model

The device speaks the modern transport, version two, with split queues. The "version one, `queue_num_max` zero" reading came from two joined causes rather than a legacy device. The QEMU `virt` net transport arrives transitional and answers version one until the driver asks for more, and the driver read `QueueNumMax` at the wrong register, so it saw a zero that belongs to an empty offset. Ask the device for the modern face with a single command-line global, negotiate the version-one feature bit, program the split-queue address registers, and set `QueueReady`. The legacy PFN vring stays declined, because the modern split layout is the OASIS 1.2 path, matches the study reference, and keeps one clean grammar for every queue.

## Question Two — The 0x400 Reading and the Padding

The value 0x400, one thousand twenty-four, is the true `QueueNumMax`, and it lives at offset `0x034` in both the legacy and the modern layouts. The confusion traced to the driver's own constants: its "modern" `QueueSel`, `QueueNumMax`, and `QueueNum` pointed at `0x028`, `0x02c`, and `0x030`, where a modern transport keeps other things, so the max read back as zero and only the legacy-offset probe found the real number. With the constants corrected to `0x030`, `0x034`, and `0x038` — the offsets both faces share — the max reads 0x400 on the modern path. The `vring_size` padding arithmetic dissolves along with the PFN vring, because the split layout addresses three separate rings and packs nothing.

The register correction, stated plainly:

| Register | Driver had | OASIS 1.2 (shared by both faces) |
|----------|-----------|----------------------------------|
| `QueueSel` | `0x028` | `0x030` |
| `QueueNumMax` | `0x02c` | `0x034` |
| `QueueNum` | `0x030` | `0x038` |

## Question Three — Both Queues Before DRIVER_OK

Each guest arms only the queue its role uses, and that serves the split-role lab: the TX guest arms queue one, the RX guest arms queue zero, and each reaches `DRIVER_OK` with its own queue live. The correctness that truly matters sits one layer down — RX and TX carry separate descriptor, available, and used pages, because sharing one ring triple across two queues corrupts both. A guest that drove both directions itself would arm both queues before `DRIVER_OK`; the lab's single-role guests arm one apiece, which is enough to cross.

## Question Four — The Command Line

The command line was missing one property, and adding it opens the modern face. Pass `-global virtio-mmio.force-legacy=false`, and the net device at slot seven presents version two with the split-queue registers and a feature word that offers version one. No bus reattachment is needed and the device stays `virtio-net-device` on the MMIO bus. With this global in place the diagnostic reads cleanly: base `0x10008000`, version two, `QueueNumMax` 0x400, status settling to `0x0b` after features acknowledge.

## Question Five — The Ruled Init Order and the Fences

The order that greens runs straight through. Reset the status, then acknowledge, then set driver, then read the offered features and write back version one together with MAC, then set features-ok and confirm the bit sticks. For the queue, select it, set its size, write the descriptor, available, and used addresses, and set it ready. Then set driver-ok, post the buffers into the available ring, publish the available index, ring the notify, and poll the used index. Two fences carry the ordering: one after the ring contents and before the available index publishes, and one after the index and before the doorbell. The deeper requirement lives in the memory itself — the whole vring is volatile, so a store the device must read is never dropped as dead and a value the device writes is never cached stale. That single discipline replaces every incidental barrier.

## Question Six — The Authoritative Source

Three sources hold three roles. OASIS virtio 1.2, section four point two, is the register and transport authority, and it settled the offsets and the init order. The QEMU transport itself is the ground truth for what this metal presents, read honestly through its own trace output rather than assumed from a reference. The xv6-riscv split-queue pattern stands as a helpful illustration held beside the specification rather than as law. And the source that named the real blocker belongs to none of these, because the fault lived in the boot layout.

## The True Blocker — a Boot That Held Only by Luck

The hang wore a virtio costume and hid a boot fault. QEMU on `virt` with `-bios none` starts the hart at the base of RAM, and the linker script never accounted for the unwind sections the toolchain emits. Under the faster optimization modes the linker laid `.eh_frame_hdr` and `.eh_frame` — unwind data, not code — at the reset address, which pushed `_start` downward, so the machine executed unwind bytes, faulted on a load near address zero, trapped into a zero vector, and fetched from zero forever. The gentler and the size-optimized builds placed those same sections after the code, so the entry stayed put and the guest ran, which is why the fault came and went with the optimization mode and why adding any console output could flip a build green. The story the console told — a poll that never completes — was true only downstream of a guest that had already crashed before it ever reached the notify.

The fix makes the boot hold on purpose rather than by luck. The layout now discards the unwind and attribute sections, folds the small-data sections into their parents so a global never lands in an unplaced region near zero, and keeps `.text.init` first at the reset address. With that in place the guest boots identically across every optimization mode, and the entry sits at the base of RAM in each one.

## What Greened

The device wire now stands on all three rungs. Sub-lap two carries the fixed pattern between two guests over the socket link, and the receiver verifies it. Sub-lap three seals the shared datagram with `wire_format` unchanged, frames it as the ethernet payload, crosses the same link, shape-casts at the edge, and opens it — the message reads back in the clear. The TX completion greens deterministically under debug, safe, fast, and small optimization, and the sealed guests build freestanding with the full crypto in place. Aurora's seed still wakes cleanly on the corrected layout, so the boot fix carries the whole stack forward without breaking the ground already laid.
