## Mission

**Steep** names the application that lets peers share large content as sealed, content-addressed facts — pieces that steep across the network until a whole is complete — using Rye OS modules rather than a BitTorrent-shaped swarm.

## Design spirit (visionary)

| BitTorrent-shaped habit | Steep direction (our vocabulary) |
|-------------------------|----------------------------------|
| Open swarm + tit-for-tat | Named peers, sealed Comlink carriage, Pond place/hold/refuse |
| Infohash as social object | Mantra/Brix digests as the name of the whole |
| Unbounded piece buffers | Tally gardens — piece windows with stated budgets |
| Tracker / DHT as discovery | Granary-shaped naming and optional Mandi listing when commerce appears |
| Protocol width first | Smallest lovable complete: one digest, one peer path, one verify fold |

Steep inherits **logic before carriage**: prove the piece catalog and verify fold in-process before any live swarm wire.

## Published-set precedent (seated `20260712.071405`)

When a named peer asks for a piece by digest, the serving side gates on *pieces explicitly offered to that peer*, never on *any digest the local store holds*. Granary’s `serve_resin_published` is the in-tree precedent for that shape.

## First-lap sketch (not opened)

When a gate opens, the first checkable lap is likely:

1. **Catalog** — Bron piece list bound to one content digest (Anvil/Granary-shaped, our tree only in fixtures).
2. **Verify fold** — reassemble or checksum pieces into the named whole; mismatch refused.
3. **One-peer fetch** — Comlink hosted (then device) carriage of one piece under Pond customs — gated by the published-set / peer-offer precedent above.

ROCm, live TB, and soak-era protocol width stay off this track.
