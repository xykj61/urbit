## What a Forge Is

A forge is a place where code lives, grows, and is shared. It holds repositories. It accepts changes. It shows history. It lets people find each other's work and build together.

The forges the world uses today speak a snapshot protocol and show one kind of history. Snapshots named by hash, pushed and pulled over a secure channel. The history is a graph of frozen moments, each linked to its parents.

Our forge may speak a compatible wire protocol — so existing tools can push and pull — yet it shows a richer history underneath. The weave. Every line that has ever been written, with its generation and its presence, in one structure that only accretes. A merge that always completes. An annotation that shows what each side did. The snapshot graph is an import path; the weave is the native record.

## The Four Layers

### 1. Mantra — The History

Mantra is the heart of the forge. Each repository holds a weave per tracked file, linked by a commit chain. The weave is the history — not a reconstruction from snapshots, yet the living structure itself.

A push arrives as content — snapshots, patches, or raw files. The forge diffs the new content against the current weave state and applies the diff as a new generation. The weave grows; the commit chain extends. Imported snapshots are kept for interoperability; the weave is the primary record.

A person who visits the forge sees the weave: every line, its age, its generation count, the merges annotated with what each side did. A snapshot view is available for those who want it; the weave is what the forge shows first.

### 2. Brushstroke — The Surface

The forge's interface is a Brushstroke application. The same immediate-mode discipline that draws a desktop window draws the forge's web surface: values in, frame out, no retained tree.

Two surfaces, one codebase:

- **Native** — a Brushstroke window on x86_64 for the forge operator. The admin surface: repository management, user keys, replication status.
- **Portable** — Brushstroke compiled to a form that runs in a browser. The public surface: weave history, diffs, merge annotations, project pages.

The Skate terminal widget renders code with the same GPU-friendly grid layout it uses on the desktop — monospaced text, syntax-aware, redrawn from values each frame.

### 3. Comlink — The Replication

A forge does not need to be alone. Through Comlink's bounded network, forges replicate repositories to each other — content-addressed, sealed, exactly-once by hash.

A group project hosted on one forge is reachable from any peer that holds a replica. Each forge is sovereign: it decides what it hosts, what it replicates, and whose keys it trusts. No central authority. The network is the peers, and the peers are the forges.

Replication is pull-based: a forge requests a repository by its content name, and the peer that holds it serves it. The weave state is the unit of replication — one structure, transferred whole or incrementally, merging cleanly on arrival because the CRDT guarantees convergence.

### 4. Pond — The Enclosure

Continuous integration runs inside Pond enclosures. A CI job is a Pond application: it receives capabilities to read the repository and write build artifacts; it receives no capability to reach the network, modify the forge, or access other projects.

The policy is a value:

```
Pond CI enclosure:
  read:   [repository source, .brix descriptor]
  write:  [build artifacts directory]
  net:    false
  mem_mb: 2048
  procs:  64
```

A build that exceeds its memory budget fails with a named error. A build that tries to reach the network meets an absent capability. The enclosure is the boundary; the policy is the stated edge.

## Identity — Keys, Not Accounts

A contributor to a forge is identified by a cryptographic key pair — the same owner-key model that runs through the whole family. A push is signed; the forge verifies the signature. A person's identity is their key, portable between forges, revocable by the owner alone.

A forge does not issue accounts. It trusts keys. A new contributor presents a key; if the repository's policy accepts it, the contributor may push. The policy names which keys may contribute, the same way a Pond policy names which paths may be read.

This means a person can contribute to forges they have never visited before — their key is their credential, carried in their own keeping, recognized by any forge that trusts it.

## Group Projects — The Social Shape

A group project is:

- **A Mantra repository** — the weave that holds every line
- **A `.brix` descriptor** — the bricks the project is made of
- **A set of keys** — the contributors, each carrying their own identity
- **A forge** — any forge that hosts the repository, or several that replicate it

Starting a group project costs nothing beyond what a person already has: a key pair, a `.brix` file, and a `mantra init`. Sharing it costs one push to a forge. Growing it costs one key added to the policy.

The forge encourages group projects by making them **cheap to begin and sovereign to keep.** A group that wants its own forge runs one binary on one machine. A group that wants to join a larger community pushes to a public forge. A group that wants both does both — the CRDT weave ensures that repositories replicated across forges always converge.

## The Path — From What Already Runs

```
[Already running]
  Mantra seed: weave + commit chain + add/log/status
  Brushstroke: designed (988), awaits display-layer study
  Comlink: sealed datagram between two harts
  Pond: policy as a value, awaits Caravan + Tally

[The forge grows from these]
  1. Brushstroke seed — one native window, text rendering
  2. Brushstroke portable — compile to browser surface
  3. Forge seed — serve one Mantra repo over HTTP, display weave history
  4. Wire protocol — accept push/pull from existing tools, update weave on push
  5. Multi-repo — serve many repositories, search, project pages
  6. Comlink replication — peer-to-peer sync between forges
  7. Pond CI — build inside enclosures, report results
  8. Key-based identity — contributor keys, signed pushes, trust policies
```

Each step is the smallest thing that runs from what already runs beneath it. The forge is not designed whole from scratch; it grows from the weave, the surface, the network, and the enclosure — each already proven in its seed.
