## Our Own Ground

This is a clean room. The display layer names only our own modules: **Brushstroke** (compositor), **Skate** (terminal), **Rishi** (keybindings and configuration), **Pond** (application enclosure), **Caravan** (supervision), **Tally** (frame memory), and **Brix** (desktop declaration). The Wayland protocol enters as a concept — a minimal, typed message set between compositor and client. Everything else is ours.

---

## Brushstroke — The Compositor

Brushstroke owns the screen. It places surfaces, dispatches input, renders frames. It speaks a minimal protocol to its clients: a client requests a surface, draws to it, and Brushstroke presents it. The protocol is typed; each extension is a small, self-contained addition.

Brushstroke does **not** manage window layout. Layout is policy; Brushstroke is mechanism. The two are separated into distinct processes, composed through a typed protocol extension. A bug in the layout manager cannot corrupt the compositor. The same isolation boundary that Caravan keeps between kernel services, Brushstroke keeps between rendering and policy.

**The frame garden.** Each redraw is a season in Tally: the `frame` garden is opened at the start of `render`, filled with the widget tree computed from current values, and cleared whole when the frame is submitted. No retained tree persists between frames. Immediate mode, made tangible by the garden.

---

## Skate — The Terminal

Skate is the terminal surface — GPU-accelerated text rendering inside a Brushstroke window. Monospaced text, syntax-aware coloring, redrawn from values each frame. Rishi runs inside Skate; Mantra's output lands here.

Skate is a Brushstroke client: it requests a surface, renders its text grid to it, and Brushstroke places it on screen. Skate is about one thing — rendering a grid of styled characters. Shell behavior, command history, and input handling belong to Rishi; Skate draws what Rishi gives it.

## Rishi — The Keybinding Language

All window management policy is declared in Rishi. A keybinding script names what each key combination does:

- **Super + direction** — move window focus
- **Super + number** — switch workspace
- **Super + Enter** — open Skate
- **Super + Space** — open the command palette
- **Super + Q** — close the focused window

The keybinding script is a value — a `.rish` file that Brushstroke reads at startup and watches for changes. Changing the desktop's behavior means editing one file. The compositor restarts the layout manager when the script changes; the layout manager reads the new bindings.

---

## Applications — Each in a Pond

Every application on the desktop runs inside a Pond enclosure. The policy names what each application may do:

- **Skate** — read/write the terminal buffer, spawn Rishi processes, access the filesystem per policy
- **File viewer** — read the allowed paths, draw a Brushstroke surface
- **Browser surface** — Brushstroke portable backend, network access per policy
- **Editor** — read/write project files, run Rishi formatting scripts

Each enclosure is bounded by Tally, isolated by Caravan, supervised with restart-on-fall. A crashing application is restarted; the compositor and every other application continue.

---

## Three Languages, No More

The desktop is configured through the same three languages that serve the whole system:

| Language | Desktop role |
|----------|-------------|
| **Brix** | Declare what exists: desktop settings, application list, display configuration |
| **Rishi** | Express behavior: keybindings, layout rules, automation scripts |
| **Rye** | Build everything: Brushstroke, Skate, the layout manager, the applications |

No window-manager-specific DSL. No theme language. No compositor scripting language. Three languages serve the entire system, desktop included.

## The Separated Architecture

```
Brushstroke (compositor process)
  ├── Renders frames from the frame garden
  ├── Places client surfaces on screen
  ├── Dispatches input to focused client
  └── Speaks typed protocol to clients and layout manager

Layout Manager (separate process, written in Rishi)
  ├── Reads keybinding script
  ├── Receives window events from Brushstroke via protocol
  ├── Decides layout (tiling, floating, fullscreen)
  └── Sends layout decisions back to Brushstroke

Skate (client process, in Pond enclosure)
  ├── Renders text grid to a Brushstroke surface
  └── Hosts Rishi REPL + Mantra commands

Other applications (each in its own Pond enclosure)
```

Each process is about one thing. Each communicates through a typed protocol. Each is supervised by Caravan. The desktop is a composition of single-stranded parts.
