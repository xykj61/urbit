## Our Own Ground

This is a clean room. We name only our own modules: **Bron** (the data notation), **Brix** (the composing language that uses Bron), **Rishi** (the shell that reads and writes Bron), **Mantra** (the history), **Comlink** (the wire), and **Pond** (the enclosure). The insight that minimal syntax reduces token cost enters as a principle from our research; the notation is ours.

---

## The Distinction

| Module | What it is | Analogy |
|--------|-----------|---------|
| **Bron** | Data notation — carries values as text | JSON, TOML, ZON |
| **Brix** | Composing language — declares what systems are made of, interfaces with Mantra, targets Rye OS syscalls through Aurora and Tally | Nix expression language |

Brix *uses* Bron as its surface syntax — the way Nix expressions produce attribute sets. Yet Bron is also used independently: Pond policies, Comlink message headers, forge descriptors, editor settings, LLM-facing configuration. Anywhere a value needs to be written down, Bron is the notation.

## The Format

Bron is plain key-value. One field per line. Space separates key from value. `#` begins a comment. Blank lines are ignored. File extension: `.bron`.

```bron
# a Pond enclosure policy
read /usr /lib /etc/ssl/certs
write /home/xy/veganreyklah2
net false
mem-mb 4096
procs 512
```

**No quotes.** A value is the text after the first space, trimmed.

**No braces.** Structure is conveyed by key names, not by nesting syntax.

**No type annotations.** A value is a string. The reader knows whether to parse it as a number, a path, or a boolean by the key's meaning.

---

## Why This Shape

### Token efficiency

Every quote, brace, bracket, comma, and colon in JSON carries no information — pure syntactic overhead. Bron carries none of them. The same data in Bron uses roughly half the tokens of JSON. This matters when an LLM reads configuration on every turn, when a forge displays metadata for thousands of projects, and when a Comlink message carries a policy across the wire.

### Human readability

A person reads a `.bron` file the same way they read a terminal listing. No formatter, no syntax highlighter, no special tooling. The structure is visible in the key names and the whitespace.

### Trivial parser

A parser for Bron is:
```
for each line:
    trim whitespace
    if empty or starts with '#': skip
    find first space: key = before, value = after
```

This fits in Rishi's `read-file` + `lines` + `starts-with` vocabulary. The parser is smaller than the format specification.

| **Comlink headers** | `content-name <hash>`, `sealed true` |
| **LLM context** | Project configuration sent to agents — fewer tokens, same information |

## Growth Path

The seed Bron is flat key-value. As use cases grow:

1. **Repeated keys** — already present in `.brix` (`file` appears multiple times). The reader collects them into a list.
2. **Sections** — a `[section]` header groups related keys. Still one field per line.
3. **Tabular encoding** — for arrays of records, a header row names the fields and data rows carry the values. The same efficiency ZON achieves for repeated structures.

Each growth step adds the minimum syntax required. Bron never gains braces, brackets, or quoted keys.

---

## Bron and the Value Model

A Bron file is a value. When Rishi reads a `.bron` file, it produces a record — the same record type that flows through every module. The record's fields are the keys; the values are strings (or lists, for repeated keys). A Bron file is how a Rye value looks when written to disk.
