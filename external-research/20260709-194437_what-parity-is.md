## In One Breath

**Parity** is this repository's big green light. You run one Rishi script from the repo root; it runs a long list of small programs called **witnesses**; each witness either proves its claim on this machine (**GREEN**) or fails (**RED**). When the whole list finishes green, the tree is fit to push. The number people say — **parity 206** as of the Jul 9 compass-rose land — is how many of those witness slots the suite currently carries (the count moves as laps land or retire).

It is not a version number for the product. It is a count of promises kept on metal.

---

## Where the Name Came From

The file is still called `tools/parity.rish`. Its header tells the older story: it began as a **differential** check — run the same tests against a "baseline" std and a "strengthened" std, and demand they agree. That differential gate retired when `rye/lib/std` became a pristine symlink to the vendored Zig toolchain. The filename stayed (accrete, never break). The living meaning grew into: **one ordered regression suite that every push must clear**.

So when someone says "parity," they usually mean three related things at once:

1. The script `tools/parity.rish`
2. The act of running it (`rishi/bin/rishi run tools/parity.rish`)
3. The count of witness stanzas it currently holds (today **206**)

## Witness, Gate, Trio — Three Different Jobs

Vocabulary law ([`one-word-one-gate`](../context/specs/20260704-020800_one-word-one-gate.md)):

| Word | Meaning here |
|------|----------------|
| **Witness** | A program that proves a claim on metal and answers green or red |
| **Gate** | A precondition only Kaeden's ruling or a landed event opens (not a test) |
| **The trio** | Three scripts that guard pushes: `parity.rish` · `parity-selftest.rish` · `additive-gate.rish` |

**Parity** is the large suite — almost everything. **Parity-selftest** guards the std symlink story. **Additive-gate** watches structural drift when `rye/lib/` changes. Together they are the witness trio. Forward prose prefers "the trio" over the older phrase "gate trio."

A **lap** is a small complete build. When a lap is done, it usually earns a new witness stanza in `parity.rish`, and the parity count goes up by one (or more). Removing a retired path can drop the count — SLC-L3 settlement removal took the suite from **198** to **197**; MALA, Sigma, WOV, dual D1, and the compass rose climbed it to **206**.

## What One Witness Looks Like

A typical hosted witness is a short `.rish` file that:

1. Builds a binary with `rye build` (often setting `RYE_ZIG` to the vendored toolchain)
2. Runs `selftest` (and sometimes `demo` or a device wire lab)
3. Asserts the output contains `GREEN` and any named unwelcome-path lines
4. Prints its own `GREEN: …` line for the suite log

Example shape (MALA M1):

```text
build → linengrow/bin/mala selftest → assert GREEN → say GREEN: MALA M1 …
```

Parity itself is a long sequence of:

```text
say "witness suite: …"
let x = run ["rishi/bin/rishi" "run" "tools/some_witness.rish"]
assert x.ok else "… failed"
say "GREEN: …"
```

The first block is special: it runs a large list of `rye/tests/*_test.rye` files against pristine std in one map — the old regression heart. Everything after that is product and discipline witnesses (SLC, Open Asks, Mantra, Caravan, TAME checks, MALA, doorway, identity, device wire, …).

## How to Read "Parity 199"

| Reading | Meaning |
|---------|---------|
| **199 witnesses** | About 199 distinct `witness suite:` stanzas / claims in `parity.rish` (the ROADMAP line lists the named product ones; the std regression block is one big GREEN for many small tests) |
| **GREEN on metal** | Someone ran the suite on a real machine and it exited 0 |
| **Wired into parity** | A new lap's witness was appended to `parity.rish` so future pushes cannot forget it |
| **Parity-eligible** | Deterministic, no live network, no external interpreter — safe to put in the suite (chain-read and live Sui verify stay on-demand) |

Rough runtime today: on the order of **two and a half minutes** with `RYE_ZIG` pointing at `vendor/zig-toolchain/zig`. Some witnesses spawn localhost UDP demos or QEMU device labs; those are the ones that can hang if a UDP race appears (the OA-L3 bind-before-send fix was exactly that class).

## What Parity Is *Not*

- Not a release version of Linengrow or Rye OS
- Not a proof that cryptography is audit-grade (a selftest proves the pattern runs; formal review is a later season)
- Not a substitute for Kaeden's **gates** (Edit 5, WOV, Mycelium at M4, lap words)
- Not required to invent a new number for every doc edit — only for new witness slots

---

## The Habit Around It

1. **Before claiming a lap finished** — its witness is green alone, then green inside full parity.
2. **Before `go` (commit + push)** — full parity green on metal when the change touches code or witnesses.
3. **When counting** — ROADMAP and TASKS carry the living number; session logs record the run.
4. **When something hangs** — clear stale QEMU/UDP children; prefer bind-before-send for multi-datagram demos; re-run clean.

On this host, many witnesses need Zig on `PATH` or `RYE_ZIG=vendor/zig-toolchain/zig` — the suite assumes the vendored toolchain the tree already pins.

## Why It Matters

Parity is how this tree keeps Gall's Law honest in practice. A complex system that works grows from simpler systems that worked — and each simpler system leaves a witness behind so it cannot silently stop working while the next lap is built. The number goes up when we keep a new promise. The suite stays green when we still keep the old ones.
