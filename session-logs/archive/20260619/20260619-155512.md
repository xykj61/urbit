## Thinking trace

1. Identified what River and Ghostty each offer, in relation to our stack:

   **River** — a Wayland *compositor* (display server) written in Zig:
   - Shows the Wayland protocol at the server side in Zig's grain: wl_display, wl_registry, surface/buffer lifecycle, the SHM pool pattern, the sendmsg/recvmsg fd-passing used by all Wayland clients.
   - Its client-facing code (how compositor and client negotiate the protocol) teaches the exact posix.socket / mmap / sendmsg / recvmsg calls that Brushstroke's Wayland backend needs.
   - River uses wlroots (a Wayland compositor library) — the wlroots bindings in Zig are the reference for how to speak Wayland from Zig code.
   - For our purposes: if we run a full Pond OS, we either use River as the compositor or grow our own. River's architecture is worth deep study before we decide.
   - On a hosted Linux desktop: Brushstroke's native backend is a Wayland CLIENT (not compositor). River shows the client protocol from the other side, which illuminates what the client must do.

   **Ghostty** — a GPU-accelerated terminal emulator written in Zig:
   - Is almost precisely what Rishi Pond needs as its first form: GPU-rendered text, a terminal state machine (VT100/ANSI), scrollback, Wayland + X11 + Metal backends behind one seam.
   - Its backend abstraction (Metal on macOS, Vulkan/OpenGL on Linux, WebGL for browser) is the model for Brushstroke's seam.
   - Its terminal cell buffer (text + color + cursor) is the model for Brushstroke's text-area widget — the most important widget for a Rishi REPL.
   - Its Wayland client code (wl_surface, wl_buffer, xdg_surface, wl_keyboard, wl_pointer) is the reference Brushstroke needs to write its native backend.
   - Ghostty uses libghostty as a library (C API) for embedding — meaning a Pond app could, in the near term, host a Ghostty surface via C bindings rather than rewriting a terminal from scratch. This is the "wear a thin costume first" move for Brushstroke.
