Is there a way to do a fast monolith TAME Rye kernel that is just as safe? Or is a microkernel design objectively safer and therefore a priority for our TAME Rye goals? ## Thinking trace

1. Identified the core question: safety trade-off between microkernel isolation (seL4/Redox) and TAME-disciplined monolith (Linux-style, but in Rye).

2. Mapped the microkernel performance history:
   - Mach (1980s): slow because copy-based IPC — 10–50× overhead over monolith.
   - Liedtke/L4 insight (1993): IPC latency is a design choice, not inherent to the model. L4 achieved ~4µs round-trip on 1990s hardware.
   - seL4 (modern): ~500 ns round-trip IPC; ~150–250 ns FastPath on ARM Cortex-A57. This is 2–5× a Linux syscall, not 10–50×.

3. Identified the zero-copy IPC solution: pass a capability to a Tally-bounded shared region rather than copying data. IPC message = capability + length; data stays in place. This is already the Aurora wire model (shared-memory mailbox) and the Wayland SHM-pool model — our own code already uses this pattern.

4. Assessed the TAME priority order as applied to this question:
   - Safety: microkernel wins decisively. TCB of ~10,000 lines (seL4) vs. hundreds of thousands in a monolith. Formal verification is tractable for the microkernel; it is not a practical target for a full monolith.
   - Performance: with zero-copy IPC via Tally-bounded regions, the performance gap for Caravan's actual use patterns (operations at 100–100,000/s, not tight inner loops) is 0.005–5% CPU overhead — acceptable.
   - Joy: capability model is cleaner — an unforgeable, transferable reference to a resource, which is exactly what Rye values are.
   - TAME conclusion: microkernel wins on the priority that comes first.

5. Mapped the design implication for Caravan:
   - Minimal trusted core: capabilities, scheduling, IPC — nothing else.
   - Caravan's kernel-adjacent layer: capability-based process management, issuing Tally-bounded region capabilities to new processes.
   - Service tree: VFS, drivers, display — each supervised by Caravan above the kernel.
   - IPC data path: Tally region + capability = zero-copy, bounded.
   - This is Pond full-OS horizon work, but the study must happen before Caravan v1 hardens so the path toward the microkernel core remains open.

6. Identified and named sources: seL4, Redox OS, Liedtke "On µ-Kernel Construction" (SOSP 1995), sel4bench.

## Decisions

- Concluded: microkernel is objectively safer by formal verification criteria; the performance cost is bounded and manageable with zero-copy IPC via Tally regions.
- Stated the design implication for Caravan plainly, without naming our modules inside the external-research piece (they are named by their concepts: "process supervisor," "capability-based process manager," "service tree").
- Noted that this is Pond full-OS horizon work — not near work, but the study must happen before Caravan's supervision design hardens.
