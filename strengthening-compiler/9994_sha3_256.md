## The Function and Its Role

`std.crypto.hash.sha3.Sha3_256` is one line in `rye/lib/std/crypto/sha3.zig`:

```zig
pub const Sha3_256 = Keccak(1600, 256, 0x06, 24);
```

It is the same generic `Keccak` type that SHA3-512 uses, with two parameters changed: `output_bits` is 256 instead of 512, and the block rate therefore widens to 136 bytes (the Keccak sponge absorbs in blocks of `(f - 2 * output_bits) / 8` bytes). Everything else — the permutation, the padding, the squeeze, the `hash`/`update`/`final` wrapper methods, and the TAME assertions added in the two prior passes — is shared.

This shared implementation means the assertions from 9998 and 9997 already cover SHA3-256. The pass 9994 strengthening is therefore confirmatory rather than additive at the code level: we confirm that the invariants hold for SHA3-256's specific block geometry, and we add the witness tests that will keep them confirmed.

## The Block Geometry Difference

The key structural difference between SHA3-256 and SHA3-512 is the rate:

| Variant   | output_bits | rate (bytes/block) |
|-----------|-------------|--------------------|
| SHA3-256  | 256         | 136                |
| SHA3-512  | 512         | 72                 |

The rate determines where the block boundary sits. The TAME assertion in `update` reads:

```zig
assert(self.st.offset <= block_length);
maybe(self.st.offset == block_length);
```

For SHA3-512, the boundary is 72 bytes; an exact-multiple input rests the cursor at 72. For SHA3-256, the boundary is 136 bytes; the same boundary case occurs at multiples of 136. The assertion's form is identical; only the value of `block_length` differs, and that is a compile-time constant fixed by the generic parameters.

Pass 9998 confirmed the invariant for 72-byte blocks with a boundary sweep. This pass confirms it for 136-byte blocks.

## The Known Answer

SHA3-256 of the message `"Rye"` (three bytes, ASCII `0x52 0x79 0x65`):

```
d10e097c85b00d288dfe8b29b8feaa8b331db1d37c3b5366c0052958779e3e40
```

Verified by running `sha3_256_test.rye` against both the baseline and Rye's std. The test renders the digest in an explicit, bounded hex loop — no volatile formatting call — and asserts the result matches the known value exactly.

The empty-string vector:

```
SHA3-256("") = a7ffc6f8bf1ed76651c14756a061d662f580ff4de43b49fa82d80a4b80f8434a
```

Confirmed in the boundary sweep test by asserting the first four bytes: `0xa7 0xff 0xc6 0xf8`.

## The Boundary Sweep

`sha3_256_boundary_test.rye` hashes eight inputs of lengths that bracket SHA3-256's 136-byte block boundary:

```
lengths = { 0, 1, 135, 136, 137, 272, 273, 512 }
```

The lengths 136 and 272 (exact multiples of 136) are the ones that rest the absorb cursor at `offset == block_length` — the boundary the strengthened assertion allows. No assertion fired on any of the eight. The empty-string known-answer held. Sink value (xor of first bytes, printed): `249`.

This sweep proves the same thing the 9998 sweep proved for SHA3-512: the bound `offset <= block_length` (not `< block_length`) is the true invariant for SHA3-256's block geometry, and it holds across inputs that straddle the boundary in every direction.

## Why This Pass Matters for Mantra

Mantra names every weave line by the hash of its content. SHA3-512 was chosen for Aurora and Comlink's identity layer — its 512-bit digest carries maximum collision resistance for long-lived identity roots. SHA3-256 serves a different role: naming the shorter-lived content of a weave, where 256 bits of collision resistance is sufficient and the shorter hex string fits comfortably in a text line alongside the content it names.

The strengthening passes 9993–9991 (mem diff primitives, `std.fs` boundary assertions, `Dir.iterate`) will add the operations Mantra needs to read and write files. When those land, Mantra's seed depends on SHA3-256 — this pass — being confirmed and guarded. The witness test is the guard; the known answer is the anchor.
